# 01. BMA — 事業・ミッション分析
    **参照:** ISO/IEC/IEEE 15288 の技術プロセスに合わせたBusiness / Mission Analysis。

    ## 1. ミッション
    `RELAY` は、複数のクラウド／ローカル推論プロバイダを統一メッセージ契約、ルーティング、ストリーミング、ツール呼び出し、監査で扱うOSSゲートウェイ。  
    目的は、対話AIの機能を「モデルへの一回限りの指示」から、**検証可能・差替可能・監査可能な基盤能力**へ切り出すことである。

    ## 2. 成功条件
    - 1日以内にローカル環境で起動し、HTTP APIと最小SDKから主要フローを実行できる。
    - 商用SaaS、オンプレミス、ゲームバックエンドへライセンス上の強い制約なく組み込める。
    - 多テナント環境で越境取得・越境更新を許容しない。
    - 外部LLM／サービスに依存する箇所はPluginまたはAdapterとして差し替えられる。
    - 主要操作が監査・再現・テスト可能である。

    ## 3. ステークホルダーと価値
    | ステークホルダー | 必要な価値 | 失敗時の影響 |
    |---|---|---|
    | 開発者 | 明確なAPI、ローカル実行、拡張性 | 独自実装が増え保守不能になる |
    | プロダクト責任者 | 状態・ルール・Versionの説明可能性 | 品質事故や意図しない挙動 |
    | 安全・法務・監査 | 安全側の失敗、証跡、削除可能性 | 監査不能・データ事故 |
    | 利用者 | 誤用されない一貫した体験 | 信頼低下、予期しない状態変更 |
    | 運用者 | 観測性、復旧、移行手順 | 障害長期化・コスト増大 |

    ## 4. 制約
    - コアはOSSとして単体で稼働し、特定のAIプロバイダやクラウドに固定しない。
    - デフォルト設定は最小権限・最小保存・tenant分離・fail-closedを優先する。
    - 文書とAPIは英語命名を主とし、説明は日本語でも読めるようにする。
    - 互換性破壊はSemVerとmigration guideで明示する。

    ## 5. 非対象
    - モデルの学習・ファインチューニングを実施しない。
- プロバイダ固有機能を無理に共通化しない。
- 秘密情報の永続保管庫を自前実装しない。

    ## 6. ビジネスリスクと対応
    | リスク | 対応 |
    |---|---|
    | 情報漏えい | Routeでデータ分類と出力先を検証し、違反時は送信前に遮断する。 |
| 二重実行 | request ID・tool call state・provider attemptを永続化する。 |
| Provider差異 | 最小共通契約とcapability discoveryを設ける。 |

    ## 7. MVP意思決定
    MVPでは管理UIや課金連携を後回しにし、API、CLI、SDK、テスト可能なコアを優先する。
